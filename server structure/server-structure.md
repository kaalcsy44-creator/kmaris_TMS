# KTMS 서버 구조 및 유료 전환 기대효과

## 1. 현재 서버 구조 (전부 무료 티어)

```
[사용자 브라우저]
       │  HTTPS
       ▼
┌──────────────────────────┐
│  Vercel  (프론트엔드)       │   ktms-web.vercel.app
│  Next.js / web/           │   · 빌드시 NEXT_PUBLIC_API_BASE 주입
└──────────┬───────────────┘
           │  JSON API (CORS: *.vercel.app 허용)
           ▼
┌──────────────────────────┐
│  Render  (백엔드)          │   ktms-admin-api.onrender.com
│  FastAPI / ktms/          │   · uvicorn 1 프로세스
│  admin_api:app            │   · JWT 로그인 + 정적 토큰
│  render.yaml 블루프린트     │   · init_db.py 부팅마다 멱등 실행
└──────────┬───────────────┘
           │  postgresql (sslmode=require)
           ▼
┌──────────────────────────┐
│  Neon  (DB)               │   neon-violet-lever
│  PostgreSQL               │   · AWS 싱가포르, main 브랜치
└──────────────────────────┘
```

| 계층 | 서비스 | 플랜 | 핵심 제약 |
|---|---|---|---|
| 프론트 | Vercel | Free (Hobby) | 상업적 사용 약관상 제한, 대역폭 100GB/월 |
| 백엔드 | Render | Free | **15분 무사용 시 슬립 → 재요청 시 ~50초 콜드스타트**, 월 750시간, RAM 512MB |
| DB | Neon | Free | 0.5GB 스토리지, 컴퓨트 자동 정지(스케일 투 제로), 191.9 컴퓨트시간/월 |

추가로 백엔드는 OCR(Anthropic), SMTP 메일 발송, 트래킹 링크 기능을 환경변수로 물고 있습니다.

## 2. 유료 전환 시 기대효과

가장 체감이 큰 병목은 **Render 백엔드의 콜드스타트**입니다.

### Render 백엔드 (가장 우선순위 ↑)
- **Free → Starter (약 $7/월)**: 슬립 없음 → **첫 접속 50초 대기 제거**, 항상 따뜻한 상태. 실사용자 피드백상 가장 큰 개선입니다.
- RAM 512MB → 더 큰 인스턴스로 PDF/OCR 동시 처리 안정화, 타임아웃 감소.
- 무료의 월 750시간 한도(여러 서비스 돌릴 때) 걱정 없음.

### Neon DB
- **Free → Launch (약 $19/월)**: 스케일-투-제로로 인한 쿼리 지연 제거, 스토리지·컴퓨트 한도 상향, 자동 백업/PITR(특정 시점 복구) → 운영 데이터 안전성 확보.
- 현재 0.5GB는 파일럿엔 충분하나, 실거래 데이터·문서 메타가 쌓이면 곧 한계.

### Vercel
- **Hobby → Pro (약 $20/월)**: Hobby는 약관상 비상업 용도라 **실제 회사 운영이면 라이선스 측면에서 Pro 필요**. 대역폭·빌드 시간 상향, 팀 협업, 분석 기능.

### 요약 권장안
| 단계 | 조치 | 비용 | 효과 |
|---|---|---|---|
| **1순위** | Render Starter | ~$7/월 | 콜드스타트 50초 제거 (체감 최대) |
| 2순위 | Neon Launch | ~$19/월 | 백업/복구 + 지연 제거 |
| 3순위 | Vercel Pro | ~$20/월 | 상업 라이선스 적법화 |

**우선 Render만 유료($7/월)로 올려도** 사용자 체감 품질이 가장 크게 좋아집니다. 실제 회사 운영(상업적)으로 쓴다면 Vercel Pro는 비용보다 **약관 준수** 차원에서 검토가 필요합니다.
